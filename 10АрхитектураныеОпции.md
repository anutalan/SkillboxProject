## 10. Архитектурные опции

Решение: Микросервисная архитектура

Обоснование: 
- Нарезать будущую платформу можно достаточно легко по бизнесовым доменам, а так как в наличии достаточное кол-во разработчиков, то можно отдельные сервисы распределить между разработчиками, в том числе по их локации, языку. 
Получаем независимые релизы, закрепление команд за разработкой определенных компонент-модулей.
- Также независимость дает возможность отдельно проводить подготовку к релизам, проверять компоненты под нагрузкой, масштабировать отдельно каждый сервис в меру своей нагруженности.

- Для большой платормы на несколько стран, на весь мир) с большим количеством интеграций между компонентами данный подход самый подходящий и тем более если физически они будут развернуты даже в разных облаках. 
Тем более уже сейчас в разных странах ,в разных облаках есть онлайн магазины, с которыми тоже надо интегрироваться

- В самом начале можно часть сервисов реализовать группой, но так, чтобы потом их было легко разделить на отдельные микросервисы.

- Для фронта использовать BFF с делением api на mobile\ web. 

- Отдельные сервисы будут использовать NoSQL базу данных, такие как трекинговые данные и данные счетчиков. Для социальных сетей нужны базы данных с графами и возможностью хранить разные документы (фото, видео - контент).
